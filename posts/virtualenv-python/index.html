Virtual Environments in Python

My time is precious

What are virtual environments?

Virtual environments in python are environments (python, pip, libraries, etc) that are isolated from the ones installed globally in your system. Since they are isolated from the ones installed in your system, you can choose a specific version of a library or framework to work with, without affecting the one installed locally.

How does it work?

When you initialize a virtual environment it creates a fresh installation of python with no installed packages for that specific environment. You can install the requirements as you go. When you’re done you can deactivate the environment to go back to your default python environment(locally installed).

Why are they important?

Consider this: you are looking for a piece of software. You found it. But, you discover that it is written in a much more older version of a python library than the one currently installed on your system. It won’t run because many things have changed and the current version no longer supports those APIs. What do you do? Do you uninstall your current version just to run a software?

No. You run it inside a virtual environment. That way, you make sure the software runs the way it was intended to, using the exact versions of the dependencies it was designed with, without making changes to any of the installed packages in your system.

Setting up a virtual environment

Note: This article assumes you are in a *nix environment and you are using a POSIX compliant shell (preferably bash or zsh). Any command not python or pip may or may not run in other shells or environment

Create a project directory.

First, we need to create a project directory (the directory where we would store our work)

1
2

mkdir example
cd example

Install the virtualenv package.

1

pip install virtualenv

Note: The package manager of your operating system might ship with an older version of virtualenv. The preferred way of installing virtualenv is with pip.

Initialize the virtual environment

1

python3 -m virtualenv venv

Or, you can specify the python version (or path) to use with the -p parameter. Like so:

1

python3 -m virtualenv -p /usr/bin/python3.8 venv

This runs the virtualenv program and creates a virtual environment inside a subdirectory named venv(you can change this).

Activate the virtual environment

1

source ./venv/bin/activate

If everything went right, you will notice your shell prompt has changed. It now contains the name of the virtual environment directory (or the name of project directory, dependending on your prompt configuration).

Installing packages inside the virtual environment

Run:

1

pip freeze  # This command lists all installed python packages

Right away you will notice there are no packages installed. Yes, that’s how it’s supposed to be. You can now start installing packages.

To install a package, follow the syntax:

pip install <package-name>==<version>

You can omit the version if you are not sure, we will talk about it in a bit. Some valid examples:

1
2
3

pip install Django==3.1.4
pip install virtualenv
pip install Flask==1.1.2

Best practice:

  • Create a requirements.txt file in the folder.
  • Write all the packages you want to install inside it, with each package starting in a new line.
  • You can choose to include the version, we can fix it easily.
  • Writing the above installed packages in requirements.txt:
1
2
3

Django==3.1.4
virtualenv
Flask==1.1.2
  • Then you can easily install all the packages with:
1

pip install -r requirements.txt

Your environment is now ready and you can start working on your project

After finishing your work, move to the folder requirements.txt exists and run this command to replace it with all installed packages along with their version:

1

pip freeze > requirements.txt

Deactivating the environment

Once you are done working, you might want to get out of the environment. Run this command:

1

deactivate

or just close your terminal if you too tired to type anything and have a nice nap.

Check the image here for reference.

How to use virtual environments

All said and done, how do you actually incorporate virtual environments in your workflow?

Let’s say you found a python project on GitHub (or any other repository) that you want to use. Chances are the project comes with a requirements.txt file. These are the dependencies required by the project to run. The version requirements mentioned in it might be different than what you have installed locally.

If you directly run pip install -r requirements.txt in this repo, there is a good chance many packages will fail to install due to conflicts. This is very common. And chances are you will get a messy list of installed packages on your system, half of which you will never use.

This is where virtual environments save you a lot of messy work and time. All you need to do is create a virtual environment, switch to it and install the packages mentioned in requirements. This will keep your locally installed packages minimal and clean. Also the project can use the exact versions of the dependencies with no issues. A win-win situation, it is. You are then free to do whatever you want with the project and when you are done you can simply deactivate the environment and call it a day.

Doing this might sound like a lot of work at first, but you get used to it pretty quickly.
